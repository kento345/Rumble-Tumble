# 一般レビュー基準

コード品質・プロダクションリスク・改善価値を判断する際に参照する。

---

## 優先順位

指摘事項はプロダクションへの影響順に並べる。

1. コアゲームプレイやプレイヤーの進行を壊すバグ
2. 一貫性のない挙動を生む状態管理の問題
3. 将来の機能追加を遅らせる脆弱な依存関係
4. フレーム安定性を損なうパフォーマンスリスク
5. 将来のリグレッションが見込まれる未検証の挙動
6. 可読性やチーム開発速度を上げる整理

> 個人的なスタイルの好みは指摘しない。スタイルに触れるのは、正確性・保守性・オンボーディング速度・将来の変更安全性に影響する場合のみ。

---

## よくあるリスク領域

### 挙動の正確性
- 境界ケース、無効な状態、実行順の競合、ガードの欠如、前提の崩れ

### 状態管理
- 複数のシステムが同じ状態を書く、隠れたグローバル状態、真のソースが不明確、キャッシュ値の陳腐化

### 制御フロー
- 巨大な条件分岐、重複したトランジション、クリーンアップをスキップする特殊ケース

### データ構造
- マジックナンバー、並列配列/リスト、暗黙のインデックス契約、未チェックのコレクションアクセス

### エラーハンドリング
- null 処理の欠如、フォールバック挙動の不在、部分的な初期化、サイレントな失敗

### パフォーマンス
- 繰り返しのアロケーション、フレームごとの重い処理、繰り返しのルックアップ、不要なシーン検索

### 保守性
- 意図を隠す命名、無関係なクラスへの機能の散在、テストしにくい結合

### 検証
- プレイテストパスの不在、高リスクロジックへのエディタ/テストカバレッジの欠如

## 良い指摘の条件

指摘はコードに根拠を持ち、具体的かつ行動可能であること。

```
場所     : ファイルパスと行番号
問題     : 曖昧な「においがする」ではなく、具体的なリスク
なぜ重要か: プロダクション上の影響
改善案   : 最も実践的な修正または調査の次のステップ。コード例や手順を含む。
```

確信が持てない場合は「〜の可能性がある」「〜と思われる」などの表現を使う。ランタイムのセットアップ、シーンの配線、手元にないコンテキストに依存する問題はとくに注意。

---

## 改善アイデアの基準

改善アイデアは「主要な指摘ではないが有益な変更」に使う。

良い改善アイデアの特徴:
- 将来の機能追加を簡単にする
- 重複コードを減らす
- 所有権の境界を明確にする
- ランタイムの挙動をデバッグしやすくする
- 高リスクなパスに対して的を絞ったチェックを追加する

**具体性を保つ**: 「アーキテクチャを改善する」ではなく「スコア更新の所有権を一つのサービスに集約する」のように書く。改善のやり方（コード例・リファクタリング手順など）も添える。
